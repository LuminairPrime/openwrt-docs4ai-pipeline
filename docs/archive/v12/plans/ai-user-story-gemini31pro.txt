From the perspective of an IDE-integrated AI coding tool, my primary objectives are to orient myself quickly, avoid wasting context window tokens on irrelevant text, and locate exact API signatures or code examples to generate accurate code.

### 1. Which files would you open first at root, and why?
1. **`llms.txt`** / **`AGENTS.md`**: I would open these first. The `.txt` and `.md` extensions are native to my processing, and these names signal that they contain instructions, routing information, and project rules explicitly formatted for me. 
2. **`llms-full.txt`**: If my context window is large enough, I might just ingest this single file immediately to get the entire project context in one tool call.
3. **`README.md`**: As a fallback to understand the human-readable purpose of the project.

### 2. Inside a module folder, what file would you open first for:
*   **a. broad subsystem understanding:** `<module>-single-file-complete-reference.md`. *   **`complete-reference-pages`**: **Yes.** It clearly implies a directory of chunked reference documentation.

*   **`index.html`**: HTML contains massive amounts of DOM token noise (tags, styling, scripts) that dilute semantic meaning and burn through my context window. I will always prefer Markdown or plain text equivalents.

*   **`llms.txt` / `AGENTS.md`**: Essential for initial orientation.

### 6. Would you prefer module folders at repository root, or under `modules/`?
**Under `modules/`.** 
When I list directory contents (via a `list_dir` or `tree` tool call), I process the output as text. If 50 different OpenWrt subsystems are sitting at the root alongside `README.md` and `llms.txt`, the directory list becomes noisy. Grouping them under `modules/` keeps the root clean and allows me to navigate hierarchically, which saves tokens.

### 8. Do you need metadata files such as `repo-manifest.json` in the release?
**Generally, no.** Unless I am equipped with a highly specific custom tool designed to parse your proprietary JSON schema, a well-formatted markdown index (`api-map.md` or `llms.txt`) is much easier for a generalized LLM to comprehend. JSON structural syntax costs extra tokens (brackets, quotes) compared to a clean Markdown list of files and their descriptions.
