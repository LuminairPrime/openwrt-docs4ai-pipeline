This document is the **100% fleshed-out, comprehensive implementation manual** for the OpenWrt AI Defect Discovery Pipeline. 

This guide provides exactly what to copy, where to paste it, how to run it, and why each piece exists. By following this guide, you will deploy a dynamic, multi-agent framework that tests how well foundational AIs understand OpenWrt, audits their mistakes, categorizes those defects, and automatically dictates what tutorials you must write next.

---

## 1. Pipeline Architecture (The "Why")
This pipeline operates on a **Debate/Audit** model using two distinct AI roles.
1.  **The Target Agent (The Generator):** Given an intentionally vague ("ignorant") prompt, this agent writes OpenWrt code. We want to see its natural, unprompted biases.
2.  **The Evaluating Agent (The Auditor):** Given the Target Agent's code, a strict set of OpenWrt architectural rules, and our Failure Taxonomy, this agent grades the code and outputs a JSON scoresheet.

**The Output:** A folder of JSON files calculating the frequency of specific OpenWrt hallucinations (e.g., "The AI used `systemd` 14 times"). This data mathematically proves *exactly* which cookbook tutorials you need to write for `openwrt-docs4ai`.

## 3. The Prompts (The "What")

The harness (provided in the next section) requires two distinct sets of prompts to function.

### A. The Target Prompts (Naive / Ignorant)
These are intentionally written like a junior developer who doesn't know OpenWrt natively. **Do not modify these to include "hints" like `procd` or `ubus`.**

*   **Test 1 (Shell/Init):** 
    `"Write a startup script that runs at boot. It needs to start '/sbin/my_daemon' and restart it if it crashes. It should also read the 'hostname' variable from a configuration file and apply it."`
*   **Test 2 (uCode/Backend):** 
    `"Write a script to list all active network interfaces by asking the system bus, and print their IP addresses in JSON format."`
*   **Test 3 (C-Daemon/Plugin):** 
    `"I am writing a C plugin daemon. Write the C code required to register a new system API module called 'my_plugin' that has one method called 'getStatus'."`
*   **Test 4 (LuCI/JS Frontend):** 
    `"Write a web page for my router's admin panel. It needs to have a form that lets the user change their firewall zone settings and save them."`

### B. The Evaluation Prompt (Auditor Instructions)
This is the "God Prompt" given to the Evaluating Agent. It defines *how* to grade the code.

```text
You are an Expert OpenWrt Code Auditor. Your job is to evaluate code snippets generated by a junior AI and mathematically categorize its failures according to our strict OpenWrt taxonomy.

You will be provided with:
1. The Original Request the junior AI was given.
2. The Generated Code produced by the junior AI.

You must evaluate the code and return a STRICT JSON object confirming whether it passed, and categorizing any errors using ONLY the exact tags from the Approved Taxonomy Library below.

APPROVED TAXONOMY LIBRARY:
- "ERR_LINUX_HALLUCINATION": The code uses standard Linux utilities (systemd, apt, ip, ifconfig, bashrc) instead of OpenWrt standards (procd, opkg, ubus, uci, netifd).
- "ERR_LEGACY_API": The code uses deprecated paradigms (e.g., Lua for frontend CBI views instead of modern LuCI JavaScript, or shell script wrapping jsonfilter instead of ucode).
- "ERR_MISSING_BOILERPLATE": The code is missing distinct mandatory OpenWrt structure (e.g., forgetting 'USE_PROCD=1', missing 'procd_open_instance', or omitting '#include <libubus.h>').
- "ERR_STATE_MANAGEMENT": The code attempts to maintain state via flat text files in /etc/ or /tmp/ instead of utilizing the dynamic 'uci' configuration system.
- "ERR_NON_C_COMPLIANT": The C daemon code uses C++ or fails to use standard OpenWrt event loops (uloop).

If the code violates multiple taxonomies (e.g., uses Lua AND writes to flat files), include ALL relevant tags. If the code is perfect OpenWrt code, "passed" should be true and the tags array should be empty.
```

## 4. The Execution Harness (The Automation Code)

Instead of manually copy-pasting code back and forth into ChatGPT windows, you must use this automated Python script. Save the following code identically as `run_evaluator.py` inside the `ai-defect-discovery-pipeline` folder.

You will execute it via: `python run_evaluator.py --target-model gpt-4o-mini --auditor-model gpt-4o`

### File: `run_evaluator.py`
```python
import os
import json
import argparse
from datetime import datetime
from openai import OpenAI
from pydantic import BaseModel
from typing import List

# 1. Define the Expected JSON Output Structure for the Auditor
class EvaluationResult(BaseModel):
    passed: bool
    failure_explanation: str
    classified_error_types: List[str]

# 2. Hardcode the Target Prompts (Our "Test Catalog")
TEST_CATALOG = {
    "shell_init": "Write a startup script that runs at boot. It needs to start '/sbin/my_daemon' and restart it if it crashes. It should also read the 'hostname' variable from a configuration file and apply it.",
    "ucode_script": "Write a script to list all active network interfaces by asking the system bus, and print their IP addresses in JSON format.",
    "c_plugin": "I am writing a C plugin daemon. Write the C code required to register a new system API module called 'my_plugin' that has one method called 'getStatus'.",
    "js_frontend": "Write a web page for my router's admin panel. It needs to have a form that lets the user change their firewall zone settings and save them."
}

# 3. Hardcode the Auditor Prompt
AUDITOR_SYSTEM_PROMPT = """You are an Expert OpenWrt Code Auditor. Your job is to evaluate code snippets generated by a junior AI and categorize its failures according to our strict OpenWrt taxonomy.
You must return your evaluation complying strictly with the requested JSON schema.

APPROVED TAXONOMY LIBRARY:
- "ERR_LINUX_HALLUCINATION": Uses standard Linux utilities (systemd, ip, etc) instead of OpenWrt standards (procd, ubus, uci).
- "ERR_LEGACY_API": Uses deprecated paradigms (Lua CBI instead of LuCI JS).
- "ERR_MISSING_BOILERPLATE": Missing OpenWrt structure (USE_PROCD=1, uloop, libubus.h).
- "ERR_STATE_MANAGEMENT": Maintains state via flat files instead of uci.
- "ERR_NON_C_COMPLIANT": Uses standard sleep/while loops instead of uloop in C.
"""

def run_pipeline(target_model: str, auditor_model: str):
    # Initialize the client (Assumes OPENAI_API_KEY is set in your environment)
    # Note: To test local models like Ollama, change base_url to "http://localhost:11434/v1"
    client = OpenAI()
    
    results = {}
    timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
    os.makedirs("results", exist_ok=True)

    print(f"Starting pipeline run at {timestamp}")
    print(f"Target Agent: {target_model} | Evaluating Agent: {auditor_model}\n")

    for test_name, prompt in TEST_CATALOG.items():
        print(f"[{test_name}] Phase 1: Generating standard response...")
        
        # Step 1: Ask the Target Agent to generate the code
        try:
            gen_response = client.chat.completions.create(
                model=target_model,
                messages=[{"role": "user", "content": prompt}],
                temperature=0.7 # Slight randomness to mimic natural prompting
            )
            generated_code = gen_response.choices[0].message.content
        except Exception as e:
            print(f"Failed to generate code for {test_name}: {e}")
            continue

        print(f"[{test_name}] Phase 2: Auditing generated code...")

        # Step 2: Ask the Auditor Agent to evaluate the generated code
        audit_user_prompt = f"ORIGINAL REQUEST:\n{prompt}\n\nGENERATED CODE TO AUDIT:\n{generated_code}"
        
        try:
            audit_response = client.beta.chat.completions.parse(
                model=auditor_model,
                messages=[
                    {"role": "system", "content": AUDITOR_SYSTEM_PROMPT},
                    {"role": "user", "content": audit_user_prompt}
                ],
                response_format=EvaluationResult,
                temperature=0.0 # Zero randomness for strict auditing
            )
            
            # The .parse method automatically guarantees the output matches our Pydantic EvaluationResult class
            evaluation = audit_response.choices[0].message.parsed
            
            # Save results
            results[test_name] = {
                "prompt": prompt,
                "generated_code": generated_code,
                "evaluation": {
                    "passed": evaluation.passed,
                    "failure_explanation": evaluation.failure_explanation,
                    "classified_error_types": evaluation.classified_error_types
                }
            }
            
            status = "PASS" if evaluation.passed else "FAIL"
            print(f"[{test_name}] Complete. Status: {status}. Errors: {evaluation.classified_error_types}\n")
            
        except Exception as e:
            print(f"Failed to audit code for {test_name}: {e}")

    # Step 3: Write report to disk
    report_file = f"results/eval_report_{timestamp}.json"
    with open(report_file, "w") as f:
        json.dump(results, f, indent=4)
        
    print(f"Pipeline finished! Full dynamic scoresheet written to: {report_file}")

if __name__ == "__main__":
    parser = argparse.ArgumentParser(description="OpenWrt AI Defect Discovery Harness")
    parser.add_argument("--target-model", type=str, default="gpt-4o-mini", help="The AI model generating the code")
    parser.add_argument("--auditor-model", type=str, default="gpt-4o", help="The strict AI model auditing the code")
    args = parser.parse_args()
    
    run_pipeline(args.target_model, args.auditor_model)
```

## 5. Workflow: Turning Defects into Documentation

Once you have run the `run_evaluator.py` script, you will find a JSON file in the `results/` folder. This is your **Reality Check Scoresheet**. 

### How to use the output:
1.  **Open the JSON file.** Look at the `classified_error_types` arrays across all tests.
2.  **Count the frequencies.** Did `ERR_LINUX_HALLUCINATION` appear 3 times, while `ERR_LEGACY_API` appeared 1 time?
3.  **Draft a Cookbook Chapter.** The highest frequency error indicates the most profound ignorance in the foundational model's latent space regarding OpenWrt. 
4.  **Write the mitigation.** If the model repeatedly hallucinates `systemd`, your next immediate task for the `openwrt-docs4ai` project is to write a highly detailed, concise Markdown document showing exactly how to map a `systemd` unit file into a `procd` init script.
5.  **Re-Test.** Feed that new Markdown file into your RAG pipeline / agent context window, and run `run_evaluator.py` again. If the test changes from `FAIL` to `PASS`, you have scientifically proven the effectiveness of your documentation.

This pipeline ensures zero wasted effort. You only document what the AI demonstrably does not know.
