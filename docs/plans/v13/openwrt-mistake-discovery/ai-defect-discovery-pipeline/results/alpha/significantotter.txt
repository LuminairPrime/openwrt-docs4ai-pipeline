### Result for Scenario 01

```bash
#!/bin/sh /etc/rc.common

START=99
USE_PROCD=1

start_service() {
    # 1. Read hostname from /etc/config/my_daemon
    # We use uci to parse the config file
    local new_hostname
    new_hostname=$(uci -q get my_daemon.settings.hostname)

    if [ -n "$new_hostname" ]; then
        # 2. Apply hostname to system via UCI
        uci set system.@system[0].hostname="$new_hostname"
        uci commit system
        # Apply immediately without reboot
        sleep 1
        /etc/init.d/system reload || true
    fi

    # 3. Start the daemon using procd
    procd_open_instance
    procd_set_param command /usr/bin/my_daemon
    
    # 4. Restart if it crashes (respawn)
    # Parameters: threshold, timeout, retry
    procd_set_param respawn ${respawn_threshold:-3600} ${respawn_timeout:-5} ${respawn_retry:-5}
    
    procd_set_param stdout 1
    procd_set_param stderr 1
    procd_close_instance
}
```

**Architecture & Libraries:**
I chose the **procd** init system, which is the standard process management framework in OpenWrt. It handles daemon lifecycle and automatic respawning much more efficiently than a manual `while` loop in a standard shell script. For configuration, I used **UCI (Unified Configuration Interface)**, as it is the native way to interact with OpenWrt settings.

**Difficulties & Debates:**
The main debate here is whether the hostname should be changed inside the daemon's init script or via a separate system script. Changing the system hostname during a service start can trigger a chain reaction (reloading networking/system services), which might cause a race condition if the daemon depends on certain network states. I added a small sleep and a system reload to mitigate this.

### Result for Scenario 07

```c
#include <libubus.h>
#include <libblobmsg.h>

/* Handler function for the RPC call */
static int my_plugin_rpc_handler(struct ubus_request *req,
                                struct ubus_object *obj,
                                struct blob_attr *attr,
                                char *method)
{
    struct blob_buf b = BLOB_BUF_INIT(0);

    // Check if the method is the one we expect
    if (strcmp(method, "get_status") == 0) {
        // Construct the JSON response: {"status": "ok"}
        blobmsg_add_string(&b, "status", "ok");
        
        // Send the response back to the ubus caller
        ubus_send_reply(req, &b.buf);
    } else {
        // Handle unknown method
        ubus_send_error(req, -22); // EINVAL
    }

    blobmsg_free(&b);
    return 0;
}

/* Example of registration (for context) */
/*
static struct ubus_method my_methods[] = {
    UBUS_METHOD_BUF("get_status", my_plugin_rpc_handler),
};
*/
```

**Architecture & Libraries:**
I used **libubus** and **libblobmsg**. These are the core C libraries used by OpenWrt for inter-process communication. `blobmsg` is used because it is a highly efficient, binary-based serialization format that is converted to JSON only when crossing the bus to a client.

**Difficulties & Debates:**
The debate in C development for OpenWrt is often between using `blobmsg` (fast, memory-efficient, but requires manual buffer management) versus manually constructing JSON strings (easier, but error-prone and slow). For a plugin, `blobmsg` is the only professional choice.

### Result for Scenario 10

**Script Content:**
```bash
#!/bin/sh

# Set the timezone to UTC via UCI
uci set system.@system[0].timezone='UTC'
uci commit system

# Optional: Ensure the symlink is updated if the system relies on /etc/timezone
ln -sf /usr/share/zoneinfo/UTC /etc/timezone
```

**Filesystem Location:**
Place this script at: `/etc/uci-defaults/99-set-utc`

**Architecture & Libraries:**
I utilized the **`uci-defaults`** mechanism. Scripts placed in this directory are executed by the system during the first boot process. Once they exit with a zero status, the system automatically deletes them, ensuring they "run exactly once."

**Difficulties & Debates:**
The difficulty is ensuring "permanence." Simply setting a variable in a shell session is not enough; one must use `uci commit` to write to the flash memory. The debate often involves whether to use `uci` or manually overwrite `/etc/timezone`. Since OpenWrt's `system` service is built around UCI, using `uci` is the safest way to prevent configuration drift.

### Result for Scenario 13

```bash
#!/bin/sh

CONFIG_FILE="/etc/my_app/config.json"

if [ ! -f "$CONFIG_FILE" ]; then
    echo "Error: Configuration file not found."
    exit 1
fi

# Use jsonfilter, the native OpenWrt utility for parsing JSON
# The '-e' flag allows us to use a JSONPath-like syntax
STARTUP_DELAY=$(jsonfilter -e '@.startup_delay' "$CONFIG_FILE")

if [ -z "$STARTUP_DELAY" ]; then
    echo "Error: 'startup_delay' key not found or empty."
    exit 1
else
    echo "Startup Delay: $startup_delay"
fi
```

**Architecture & Libraries:**
I used **`jsonfilter`**. While many developers reflexively reach for `jq`, `jsonfilter` is a lightweight utility specifically designed for OpenWrt/BusyBox environments to parse JSON without the heavy footprint of `jq`.

**Difficulties & Debates:**
The debate here is almost always `jq` vs `jsonfilter`. `jq` is far more powerful (allowing complex transformations), but it is often not included in minimal OpenWrt builds to save space. For a simple "read one key" task, `jsonfilter` is the "native" and more efficient choice for embedded systems.

### Result for Scenario 16

```bash
#!/bin/sh

IP1="10.10.10.2"
IP2="10.10.10.3"

# Function to run ping and prefix lines
# We use a subshell and a loop to prefix each line
run_ping() {
    local target=$1
    ping "$target" | while read -r line; do
        echo "$target: $line"
    done
}

# Run both in the background
run_ping "$IP1" &
PID1=$!

run_ping "$IP2" &
PID2=$!

# Wait for both background processes to complete
wait $PID1 $PID2
```

**Architecture & Libraries:**
This uses standard **POSIX shell (ash)** features. I used **backgrounding (`&`)** to achieve parallelism and a **`while read` loop** to intercept the stdout of the `ping` command to inject the prefixing logic.

**Difficulties & Debates:**
The primary difficulty is that the default OpenWrt shell (`ash`) does not support "Process Substitution" (e.g., `while read line; do ...; done < <(ping...)`), which is a Bash-ism. To keep the script compatible with the lightweight `ash` shell found on most routers, I had to use a helper function and standard pipes. Another debate is whether to use `tail -f` or a loop; a loop is much more robust for real-time line-by-line processing and prefixing.
