**Claude**

Good question — and honest answer: the `release-inputs/release-include/` recommendation in the plan is the weakest part. Let me lay out the real tradeoffs.

## How the overlay mechanism actually works

`release-inputs/release-include/` is applied in **stage 07**, *after* stages 05a, 05b, 05c, 05d, and 06 have already run. Stage 06 generates `llms.txt`, `llms-full.txt`, and per-module routing indexes. Stage 07 applies overlays and materializes the support tree.

This means: **any cookbook content placed in `release-inputs/release-include/` would be invisible to stage 06**. It would not appear in the root `llms.txt`, `llms-full.txt`, or any routing index. Stage 08 validation would see the files exist (overlay wins on naming conflicts) but the routing surfaces would silently omit the entire module.

That is a real pipeline integration failure. The overlay mechanism was designed for non-indexed static files like `.nojekyll` — not for primary content modules that need to participate in routing index generation.

## The existing pipeline pattern

Every existing module follows this flow:

```
L0 (clone)  →  02* extractor  →  L1-raw/{module}/  →  03 normalize  →  L2-semantic/{module}/  →  05a assemble  →  release-tree/{module}/
```

Stages 06, 07, and 08 then iterate over whatever landed in the output tree. They are mostly module-agnostic — they don't hardcode module names; they discover modules by directory listing. If `cookbook` is already present in the output tree before stage 06 runs, it gets included automatically.

## The correct approach

Store hand-authored source files in a dedicated repo directory, not in `release-inputs/`. The simplest option:

```
content/cookbook/          ← repo-controlled source, not pipeline scratch
  chunked-reference/
    era-guide.md
    common-ai-mistakes.md
    ...
```

Stage `04b` reads from `content/cookbook/`, validates/processes the files, and writes to `{WORKDIR}/cookbook/` — exactly like `02*` extractors write to `{WORKDIR}/L1-raw/{module}/`. From that point, stages 05a, 06, 07, and 08 see `cookbook` as a normal module.

**Pros of this approach:**
- Stage 06 sees `cookbook` and includes it in routing indexes automatically
- Stage 08 validates the module normally
- No timing dependency on overlay application order
- The content participates fully in the pipeline (token counting, bundled-reference assembly, HTML indexing)
- Future maintainers see a clear and consistent pattern

**Cons / what it costs:**
- One new top-level directory (`content/`) to explain in `ARCHITECTURE.md`
- `04b` must actually generate `map.md` and `llms.txt` (it can't rely on a pre-baked overlay)
- Stage `05a` must handle a cookbook module that has no L2-semantic source — it needs to assemble `bundled-reference.md` from files that arrived differently than other modules did

## Why the plan said `release-inputs/`

The `release-inputs/release-include/` shortcut looked appealing because it requires zero modifications to stages 05a, 06, 07, or 08 — you'd just drop files in and they appear. But the timing problem above kills that benefit entirely for the routing surfaces. The only scenario where the overlay approach would work correctly is if the cookbook module had no need to appear in `llms.txt` or any routing index, which defeats the entire point.

## Verdict

The plan's `release-inputs/` recommendation should be replaced with a `content/cookbook/` source directory read by `04b`. The tradeoff cost is a slightly more complex `04b` script (it has to generate `map.md` and `llms.txt`, not just copy files), but the pipeline integration is correct and the pattern is consistent with how every other module works.

| Option | Approach | Scripts to modify | Rename needed? |
|--------|----------|-------------------|----------------|

| B | Direct L2 injection via `04b` | 5+ (rename cascade from `04`→`04a`) | Yes |
| C | Early overlay application | 3 (06, 07, 08 all need module-discovery changes) | No |
| D | Full pipeline renumber to 1x/2x/3x/4x | 30-50 files | All of them |

**Three source location options:** `content/cookbook/` (recommended), `templates/cookbook/`, `.github/scripts/cookbook-source/` (not recommended).
